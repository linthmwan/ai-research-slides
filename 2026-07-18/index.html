SearchOS-V1

Towards Robust Open-Domain Information-Seeking Agent Collaboration

Yuyao Zhang*, Junjie Gao*, Zhengxian Wu, Jiaming Fan, Jin Zhang, Shihan Ma et al. · Renmin University of China + Ant Group
arXiv 2607.15257v1Announced 2026-07-16Code + project page
Source: arXiv 2607.15257v1, cover p.1
TL;DR

Search agents do not only need more browsing; they need an operating state that remembers coverage, evidence, failures, and unfinished work.

SearchOS turns open-domain search into relational schema completion with citations, then runs multi-agent collaboration over explicit SOCM state, middleware, and skills.
Abstract + §1 pp.1-3
Problem

Long-horizon search breaks when progress lives only in the prompt

  • Interaction histories bury evidence and unresolved gaps.
  • Failed search paths repeat across turns and across workers.
  • More agents can duplicate effort or sit idle at synchronized batch barriers.
  • The missing abstraction is system-maintained search state.
State should be maintained by the system rather than repeatedly inferred from conversation history.
§1 pp.2-3
Contributions

SearchOS decomposes reliability into four infrastructure layers

  1. Relational schema completion with grounded citations.
  2. SOCM: Frontier Tasks, Evidence Graph, Coverage Map, Failure Memory.
  3. Pipeline-parallel orchestration over unresolved coverage gaps.
  4. Middleware-governed execution plus hierarchical search skills.
§1 contributions p.3
Formulation

Search becomes relational schema completion with citations

Task T = (q, S), with S = {Tm, R}; output O = {(Em, Ym, Cm)}
  • Tables define attributes and primary keys; foreign keys express relations.
  • The system discovers entity rows Em and fills value matrix Ym.
  • Citation matrix Cm stores source URL plus anchored excerpt for each populated value.

Evidence + coverage

Evidence is cell-bound, not page-bound

Evidence Graph

  • Stores atomic findings rather than page summaries.
  • Records value, URL, span, schema binding, confidence, provenance tier, status.
  • Rejected or superseded nodes remain for audit.

Coverage Map

  • Each schema cell is missing, filled, uncertain, or unreachable.
  • Primary evidence is chosen by provenance, schema alignment, and authority-adjusted confidence.
  • Conflict is marked rather than overwritten.
§3.1.2-3.1.3 pp.5-6, Eq.6-8
Failure memory

Dead ends become shared operational data

  • Failure records include type, task signature, corrective guidance, recurrence count, and latest occurrence.
  • Patterns include repeated queries, inaccessible sources, failed skills, dropped branches, rejected claims.
  • Recurrence strengthens do-not-retry guidance and can seed future runs.
§3.1.4-3.1.5 p.6
Scheduling

Continuous dispatch applies pipeline parallelism to search work

Dt = Topmin(b_t, |R_t|)(Rt; p); after completion, update SOCM and refill the slot.
  • The orchestrator prioritizes schema gaps; explore/search/writer roles use scoped toolsets.
  • Ready tasks are dependency-aware Frontier Tasks.
  • Stragglers no longer block the next coverage gap from starting.

Middleware invariants

Grounding and stall control are enforced outside prompts

Evidence acceptance

Et+ = {z | Bind(z,S)=1 and Anchor(z,ot)=1}

Only schema-bound, span-anchored candidates update G and C.

Stall sensor

stall when sum Δcoverage = 0 and sum Δevidence = 0 over window w

Actions: continue, correction, backfill, drain-only, or stop branch.

Skills

SearchOS-V1 ships 280 pre-built hierarchical skills

  • Orchestrator skills: schema/row alignment and synthesis validation.
  • Strategy skills: reformulation, enumeration, disambiguation, multi-hop reasoning, stall recovery.
  • Access skills: source-specific extraction; some become typed executable tools.

Engineering takeaway

For serious agents, state is product architecture, not prompt decoration.

SearchOS is strongest where it makes hidden intermediate artifacts inspectable: gaps, provenance, conflicts, failures, budget pressure, and reusable procedures.

Appendix · Highlight 1

Why this is an OS problem

These failures arise because conventional agents treat plans, progress, evidence, and failures as transient conversation content.

這篇的核心判斷是：失敗不是因為模型不會搜尋，而是因為計畫、進度、證據與失敗都被放在短暫對話裡。§1 p.2

This state resides outside agent conversations and uses locked read-modify-write updates.

SOCM 把狀態移出對話，並用鎖定的讀改寫更新，避免多 agent 看到互相不一致的搜尋世界。§3.1 p.5

Appendix · Highlight 2

Why middleware and skills matter

Because the harness enforces these transitions outside prompts, the same safeguards apply across models and roles.

中介層把停滯偵測、預算與修正動作移到 prompt 之外，因此安全閥不依賴某個 agent 自己記得遵守。§3.3 p.8

The quality gains therefore come with less exploratory trial and error rather than additional browsing.

skill 的收益不是暴力多查，而是減少試錯與重複瀏覽，這對長程 search agent 的成本很關鍵。§5.4 p.13
