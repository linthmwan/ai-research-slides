TL;DR

第一個同時評估「單輪精度」與「多輪一致性」的指令式影像編輯基準——每個維度配專屬指標（不是一把 CLIP score 套全部），再用三個異質 VLM 評審官交叉驗證，證明量到的是「任務」而非「評審官的語言偏好」。

問題 · 前人缺陷

指令式影像編輯（IIE）的真正瓶頸不是模型，是「怎麼評」

IIE：用自然語言指令編輯影像（如「在背景加上夕陽」），由 diffusion + LVLM 驅動（InstructPix2Pix、MagicBrush、MGIE、SmartEdit…）。三類既有評估法全有硬傷：
  • ① 單一指標（CLIP-Score / PSNR / SSIM / LPIPS）：一個數字蓋不住多樣編輯——CLIP-Score 擅長高階語義卻對低階去噪無感；PSNR 擅長保真卻對物件移除無用。
  • ② 使用者研究：能抓人類偏好，但昂貴、不可複現。
  • ③ 既有基準：TedBench 僅 100 圖、EditBench 只做 mask-guided、EditVal 只有尺寸／位置可自動評；MagicBrush / Emu Edit 仍倚賴傳統 L1/L2/CLIP-I/DINO。
  • 致命盲點：幾乎全部只評單輪編輯，忽略真實使用中常見的多輪迭代編輯。

核心貢獻 · 4 根支柱

I2EBench2.0 把「評估」本身當成研究對象來設計

  1. 單輪＋多輪同時評：單輪量精度，多輪量一致性／穩定性。16 個單輪維度 ＋ 7 個多輪維度。
  2. 廣覆蓋準則：16 維度均分為 8 個高階（語義）＋ 8 個低階（修復）。
  3. 對齊人類判斷：每個準則都做使用者研究，基準分數與人類評分強相關。
  4. 研究導向洞察：逐維度拆解強弱項 → 直接指導資料／訓練／架構選型。
對圖像工程師：這不是又一個編輯模型，是一把「選對 backbone」的量尺——告訴你哪種編輯該用哪個模型。

方法核心 · 量身指標

關鍵一步：16 個維度，16 種評估法（而非一把 CLIP 套全部）

高階 8 維（語義）
  • Counting / Direction / Object Removal / Object Replacement / Background Replacement / Color Alteration → GPT-4V VQA（標註者寫「問題＋標準答案」，如「鞋架上有幾隻貓？」→「一隻」）
  • Style Alteration → 與「an image with ··· style」的 CLIP 相似度
  • Region Accuracy → 把編輯區塗白後算 SSIM（高＝非目標區被保留）
低階 8 維（修復）
  • Deblurring、Haze / Lowlight / Noise / Rain / Shadow / Snow / Watermark Removal
  • 全部用 SSIM 對比 ground-truth 修復影像
  • → 低階維度有客觀 GT，量得準；高階維度靠 VQA 抓語義對錯
設計哲學：用對的尺量對的事——語義編輯問 VQA，像素修復算 SSIM，風格轉換算 CLIP。

方法核心 · 多輪評估

多輪編輯：2–5 輪迭代，且「每一輪都得成功」

  • 多輪鎖定高階維度（除 Region Accuracy），每個樣本 2–5 輪連續編輯。
  • 嚴格計分：每一輪都必須成功；任一輪失敗 → 整個多輪樣本判定失敗。
  • 這個 gate 刻意嚴苛——它把「單發看起來不錯、迭代就崩」的脆弱性逼出來。
  • 對照組：同樣的高階維度在單輪下表現，量出 single→multi 的衰退幅度。
「每輪必過」是本基準最狠的設計：真實工作流是迭代的，單發 SOTA 不等於迭代 robust。

人類對齊 · 協議

基準分數真的追蹤人類感知嗎？逐維度驗證

協議
  • 評審者看「指令 T ＋ 原圖 ＋ M 個模型輸出」並排名
  • Rank-1 給 M 分、rank-k 給 M−k+1 分、最後一名給 1 分
  • 8 個模型 → 分數範圍 1–8，共 N×16×2 次比較
結果
  • 基準 rank 分數與人類評分在每一個維度都強正相關
  • ρ=0.98 Deblurring、多數維度 ρ≈0.94
  • → 證明這把尺量到的是人會認同的「好編輯」

★ 機制皇冠頁 · 為什麼可信

擔心「基準只是在量 GPT 的語言偏好」？他們把全部模型在三個架構迥異的評審官（GPT-4V / Qwen3VL-8B / LLaVA-1.5）下重跑一遍——模型排序穩定、維度難度模式一致、多輪一律掉分。結論對異質評審官 robust，不是 GPT 家族偏見。

可偷的方法論：你自己做任何 VLM-as-judge 評估時，換 ≥2 個架構不同的評審官、報 Max Diff——證明你量的是任務，不是評審官的方言。
§4.1.3–4.1.5 p11–18 · Table 8 · Fig.7
難度模式 · 跨評審官一致

哪些維度難、哪些易——換誰來評都一樣

  • 最難：Counting（數數）與 Object Removal（物件移除）——需要精準定位＋語義理解。
  • 最易：Background Replacement 與 Color Alteration——全域性改動，不需精確局部定位。
  • 多輪一律掉分：高階維度 single→multi 在每一個評審官下都顯著衰退。
  • 難度排序在三個評審官間保持一致 → 難度來自任務本質，不是評審官 artifact。
§4.1.3 p13–16
工程啟發 · 選型地圖

沒有模型贏在每一處——按編輯類型選 backbone

誰擅長什麼
  • 低階修復（去噪/去霧/去雨…）→ InstructDiffusion（原始指令下 4/7 低階維度最高）
  • 高階語義 ＋ 多輪（數數/方向/物件/顏色）→ Qwen-image 主宰
  • watermark/rain removal ＋ 語義高階 → FLUX Kontext 競爭力強
  • MagicBrush / InstructAny2Pix 在特定高階維度各有所長
指令穩健性鴻溝
  • 改用「多樣」（改寫）指令時，Object Removal 對 InstructPix2Pix / HIVE / InstructDiffusion / MagicBrush 直接崩潰
  • 帶 LLM/MLLM 指令編碼器的模型（Qwen-image、FLUX）反而穩定甚至更好
  • → 語言理解能力 = 泛化能力
§4.1.1–4.1.2 p11–12
工程啟發 · 多輪與美學

兩個被忽略的真相：多輪本質很難、累積編輯侵蝕美學

  • 多輪本質困難：所有維度 single→multi 大幅衰退；「每輪必過」的嚴格計分把脆弱性曝光——現有架構沒為迭代編輯最佳化。
  • 美學會被多輪侵蝕：美學分數（AP，CLIP+MLP predictor）在多輪下比單輪明顯下降——累積編輯逐步破壞視覺完整性。
  • 美學隨維度差異大：Snow Removal > Deblurring；不同編輯類型的美學天花板不同。
  • 類別效應：Scenery / Global 編輯分數最高（全域改動不需精確定位）；Animal / Human / Object 較難。
真正的 open problem：今天的 IIE pipeline 都是單發最佳化，迭代編輯的穩定性是下一個前沿。
§4.2–4.3 p19–22 · §5 p20 · Tables 7,9,10
個人解讀 · 給圖像工程師

三個可以直接帶走的東西

  1. 它是量尺不是模型，價值在「選型」：要上 IIE 功能，逐維度拆解告訴你該選哪個 backbone——低階修復選 InstructDiffusion，語義／多輪選 Qwen-image / FLUX Kontext。
  2. 偷它的跨評審官設計：自建任何 VLM-as-judge 評估，換 ≥2 個架構不同的評審官並報 Max Diff，證明你量的是任務不是評審官方言。
  3. 盯住多輪 ＋ 美學衰退：「每輪必過」gate ＋ AP 下降是真實開放問題——迭代編輯穩定性，是下一個值得投入的方向。
