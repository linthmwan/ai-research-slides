VikingMem:
A Memory Base Management System for Stateful LLM-based Applications

Jiajie Fu, Junwen Chen, Mengzhao Wang, Aoxiang He, Maojia Sheng, Xiangyu Ke, Yifan Zhu, Yunjun Gao
PVLDB / VLDB 2026 acceptedarXiv 2605.29640OpenViking trending repo2026-07-12
Source: OpenViking README lines 706-712; arXiv:2605.29640; PDF p.1
TL;DR

Long-term LLM memory is treated as a database problem: Events become logs, Entities become materialized state, and retrieval becomes a latency-bounded query plan.

The important move is not merely "better vector search". VikingMem changes the unit of memory, lifecycle, and ranking objective.

Abstract p.1; §2.2.2 p.4; §3 p.5-7
Problem

Finite context windows are the wrong abstraction for state

  • Persistent assistants, agents, education, and recommendation systems need continuity across many sessions.
  • Ad hoc logs, shallow caches, and task-specific heuristics lead to stale/conflicting memory and custom reconciliation logic.
  • Simply stuffing more history into prompts is costly, latency-sensitive, and still lacks lifecycle operations like consolidation, forgetting, and provenance.
Memory needs a service-grade substrate, not a longer prompt.
§1 p.1
Industrial observations

Three pressures shape the system design

  1. Low signal-to-noise streams: meetings, chat, logs, and trajectories contain sparse value with interleaved topics.
  2. Dynamic state: users, tools, and agent procedures evolve; insert-and-retrieve RAG accumulates contradictions.
  3. Fragmented architectures: companion chat, education, search, and agent SOPs need different memory schemas but should share infrastructure.
§2.1 p.3
Contributions

From memory prompt tricks to Memory Base Management System

  1. Defines Memory Base: selective extraction + statefulness/evolution + generalizable design.
  2. Implements VikingMem on VikingDB: extraction, management, retrieval, event/entity stores, and keyword graph.
  3. Shows deployment across five industrial categories at ByteDance, including Agent Memory.
  4. Reports SOTA long-term memory effectiveness with sub-second retrieval latency.
§1 p.2; §3 p.5; §4 p.7-8; Table 1 p.9
Core abstraction

Events are logs; Entities are materialized views

  • Event: timestamped episodic unit selectively extracted from raw sessions.
  • Entity: persistent evolving state such as user profile or tool SOP.
  • Operator: application-defined rule controlling how Events update Entity fields.
Figure 1 p.4; §2.2.1 p.3-4
Database lens

entity := SELECT OP(event.content)
FROM Events WHERE filters(event)
GROUP BY keys(event)

Changing keys, filters, and OP materializes user profiles, tool memories, SOPs, learning progress, or recommendations from the same Event log.

§2.2.2 p.4
System pipeline

Extraction -> Management -> Retrieval

VikingDB stores records and indices; VikingMem owns schema-driven extraction, deduplication, event-to-entity updates, and reranking.
Figure 2 p.5; §3 p.5
Memory extraction

One-pass extraction compiles schemas into a single prompt

  • Prior systems use separate hard-coded prompts for each memory type, causing N passes for N types.
  • VikingMem compiles Event and Entity schemas into one prompt so the LLM extracts all defined memories in one inference pass.
  • The fixed prompt prefix and memory schema can benefit from prefix-cache reuse.
§3.1 p.5-6; Figure 3 p.6
Fast entity updates

EUA turns semantic entity maintenance into patch application

  • For string fields, the extractor emits compact SEARCH/REPLACE-style patches.
  • EUA locates the best approximate span by edit distance and applies the replacement deterministically.
  • Only a small top-5 relevant entity candidate set is included in the prompt for patch generation.
E_new[f] = E_old[f][0:i] + REPLACE + E_old[f][j:]
§3.1 Algorithm 1 p.6
Segmentation

IMSM fixes the unit of memory before retrieval ever starts

  • Phase 1: semantic saliency filtering removes greetings, fillers, and low-value spans.
  • Phase 2: event-centric partitioning returns coordinate tuples for coherent segments.
  • Non-contiguous same-topic fragments (A1 + A2) can be merged while interleaved noise (B) is excluded.
This is the paper’s strongest mechanism: better memories are created at ingestion, not rescued at query time.

§3.2 p.6-7
Retrieval scoring

Recall uses relevance, freshness, and business priors

Sfinal = (1 - wtime - wbusi) Sorigin + wtime Stime + wbusi Sbusi
  • Primary path: dense+sparse hybrid retrieval over the memory store.
  • Temporal score gives a freshness tolerance window, then exponential decay.
  • Business score can be type-level or instance-level, extracted into schema fields.
  • Auxiliary keyword graph covers indirect low-similarity memories such as nicknames.
§3.3 p.7
Reranking

Late interaction is the latency-aware alternative to cross-encoders

  • Cross-encoder rerankers can push p99 into seconds, too slow for interactive memory recall.
  • VikingMem precomputes ColBERT-style contextualized vectors during extraction.
  • Quantization and token-merge compression keep overhead comparable to dense vectors.
The system pays extra compute when memories are ingested, then keeps query-time rerank fast.
§3.3 p.7
Application

Agent Memory: tool traces become reusable operational knowledge

  • Raw execution traces are verbose and noisy; VikingMem extracts salient tool invocations as Events.
  • Each Event records tool, usage situation, task goal, and success status.
  • Tool Entity evolves into lessons learned: suitable use cases, failure cases, and future suggestions.

Limitations

What this deck should not overclaim

  • Main scores rely on LLM-as-a-judge; F1 helps, but evaluation is still benchmark/protocol dependent.
  • Time weighting is disabled in the public fact-based benchmark setup, so recency/business priors are not fully stressed there.
  • OpenViking is an open-source subset of VikingMem, not necessarily the full commercial ByteDance deployment.
  • Semantic extraction and LLM_MERGE inherit model/prompt quality, even if EUA and statistical operators reduce unnecessary calls.
§5.1.5 p.10; §5.7 p.12; §3.1-3.2 p.5-7; README selection note
Engineering takeaways

For agent builders, the design lesson is surprisingly concrete

  1. Define memory schemas before collecting memory: otherwise the system only stores logs.
  2. Separate episodic Events from evolving Entity state, and make update operators explicit.
  3. Spend LLM calls at ingestion when they reduce future query noise; keep query-time recall/rerank latency bounded.
  4. Ablate segmentation separately from retrieval. If memory units are wrong, a better retriever only ranks bad evidence.
