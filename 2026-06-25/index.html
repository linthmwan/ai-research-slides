FlashAttention-4

Algorithm and kernel pipelining co-design for asymmetric Blackwell scaling
Ted Zadouri · Markus Hoehnerbach · Jay Shah · Timmy Liu · Vijay Thakkar · Tri Dao
MLSys 2026 paper indexarXiv:2603.05451v1Announced 2026-03-05
Source: arXiv p.1; MLSys 2026 papers page. Deck generated 2026-06-25 Asia/Taipei.
TL;DR

Blackwell made tensor cores too fast for the old bottleneck story.

FA4 is not just another attention kernel. It is a roofline response: shift work away from MUFU / SMEM bottlenecks, overlap softmax with async MMA, and exploit TMEM + 2-CTA MMA.

Abstract p.1; §1 p.2; §6 p.16
Why this paper today

The useful question is no longer only: how fast is matmul?

  • B200 doubles BF16/FP16 tensor core throughput vs H100: 2.25 PFLOPS vs 1 PFLOPS.
  • But MUFU exponential throughput and shared-memory bandwidth lag behind or stay flat.
  • The paper reports typical Blackwell attention bottlenecks where SMEM traffic and exp ops exceed MMA compute by 25-60%.
  • So FA4 treats attention as a co-design problem across algorithm, kernel pipeline, memory hierarchy, and scheduling.
§1 p.2 lines 68-80; §2.2 p.4 lines 190-208
Problem framing

Hopper-era FlashAttention assumptions do not transfer cleanly

Old success mode

  • Fuse attention to avoid global-memory materialization.
  • Tile Q/K/V so softmax can be computed online.
  • Exploit Hopper async execution and warp specialization.

Blackwell friction

  • Accumulator moves from registers to TMEM.
  • MMA tile grows to 128x128.
  • Non-MMA work becomes visible after tensor-core scaling.
  • Hopper MMA instructions are not forward-compatible.
§1 p.1-2; §2.2 p.3-5
Contribution map

FA4 attacks four bottleneck classes

  1. Pipeline: overlap larger-tile async MMA with softmax / memory movement.
  2. Exp: partially emulate 2x via polynomial FMA/ALU path.
  3. Online softmax: skip rescaling unless max shift exceeds tau.
  4. Backward: use TMEM + 2-CTA MMA to lower SMEM traffic and halve dQ atomics.
  5. Framework: write the kernel in CuTe-DSL / Python for fast JIT iteration.

Forward pipeline

Ping-pong two tiles: while one MMA runs, the other softmaxes

  • Two softmax warpgroups process full rows and avoid inter-warp row-max shuffles.
  • A correction warpgroup handles output rescaling outside the critical path.
  • P / S partitioning in TMEM is chosen to start by computing two S tiles immediately.
Fig.1 p.6; §3.1.2 p.6-7
Mechanism crown · forward

TMEM turns register pressure into pipeline room.

Hopper accumulators lived in registers; Blackwell accumulators live in tensor memory. FA4 spends that new memory level to decouple MMA, softmax, and correction work.

§2.2 p.3-4; §3.1.2 p.6-7
Exp emulation

Do not wait only for MUFU: split 2x across ALU/FMA

2x = 2floor(x) · 2x - floor(x)
  • Integer part: manipulate IEEE-754 exponent bits.
  • Fractional part: polynomial approximation evaluated with FMA.
  • Only 10-25% of entries are emulated, avoiding register spills from doing this everywhere.

Online softmax

Conditional rescaling removes vector work without losing final correctness

Rescale only if mj - mj-1 > tau, with tau = log2(256) = 8.0
  • Standard online softmax rescales old output whenever the running max increases.
  • FA4 tolerates slack and records statistics, then renormalizes with the true final max and denominator.
  • Warp divergence is avoided by rescaling when any thread in the warp needs it.

Backward has five MMAs and a worse SMEM story

2-CTA dQ

Distributed shared memory repacks dS for paired MMA

  • Each CTA stages half of operand B and owns its accumulator slice.
  • For dQ, DSMEM exchange gives each CTA M/2 rows and the full 2N reduction.
  • Each CTA writes half of the dQ tile, halving global atomic reductions.
Fig.3 p.11; §3.2.3 p.12
Scheduling

Causal attention is triangular, so FIFO scheduling is the wrong instinct

Throughput path

  • Use LPT: longest-processing-time-first over tile order.
  • Keep batch outermost and swizzle heads to protect L2 locality.
  • Reported H200 gains: 4-8% MHA, 7-14% MQA 8.

Deterministic path

  • Semaphore locks serialize global reductions.
  • CTA swizzling reduces waiting.
  • For causal masking, SPT order avoids first-write stalls.

Evaluation isolates attention throughput across sequence / head regimes

Backward speedups validate the 2-CTA / TMEM story

Deterministic backward

Reproducibility is treated as a scheduling problem, not just a lock

  • Naive serialization can stall CTAs badly.
  • Swizzling and SPT reduce lock waiting.
  • Reported deterministic backward reaches up to 75% of nondeterministic 1-CTA speed.

Engineer’s interpretation

FA4 is a reminder that every hardware generation changes the definition of “obvious.”

A kernel that was IO-aware on A100 and async-aware on H100 must become non-MMA-aware on B200. The invariant is not a particular trick; it is measuring the current bottleneck honestly.

Synthesis from §1, §3, §6
Practical playbook

If you build attention variants on Blackwell, borrow the questions

  1. Can softmax / exp be overlapped with at least two independent MMA waves?
  2. Can a subset of special-function work move to FMA/ALU without register spills?
  3. Does the backward roofline say SMEM, atomics, or exp is actually limiting?
  4. Does causal / varlen scheduling start with heavy tiles, or end with stragglers?
  5. Is compile time short enough that kernel design can iterate experimentally?
§3.1-3.3; §4
Limits / caveats

Do not over-generalize the win

  • Full design relies on Blackwell features: TMEM, fully async MMA, and 2-CTA MMA.
  • Polynomial exp emulation is partial by necessity; doing all entries risks register pressure and spills.
  • cuDNN incorporated some FA4 techniques after the initial release, so the exact speed gap is version-sensitive.
  • The paper’s hardware labels deserve careful reading: main text says B200, appendix A.1 says B100 180GB SXM6 for speed benchmarking.

The bottleneck shift in the authors’ words

Although Blackwell B200 doubles the tensor core throughput compared to Hopper H100 (2.25 PFLOPS vs. 1 PFLOPS for FP16/BF16), other functional units ... scale more slowly or remain unchanged.

Two small details that make the method believable

The degree-3 polynomial matches hardware to within 1 BF16 ULP on 99% of inputs.

因為最後要捨入到 BF16，低階 polynomial exp 的誤差多半被 BF16 量化誤差吃掉。§3.1.3 p.8

FlashAttention-4 reduces compile time by 20-30x compared to FlashAttention-3.

Bottom line

FlashAttention-4 is a clean example of hardware-aware algorithm design: measure the shifted roofline, then redesign the kernel around the new limiting resource.
