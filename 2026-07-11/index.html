Who Broke the System? Failure Localization in LLM-Based Multi-Agent Systems

Yufei Xia, Anjun Gao, Yueyang Quan, Zhuqing Liu, Minghong Fang · University of Louisville / University of North Texas
COLM 2026 conference paperarXiv:2607.07989v1Announced 2026-07-08
Source: cover p.1; arXiv:2607.07989v1
TL;DR

把 multi-agent debug 從「看誰看起來錯」改成「驗證誰在最早 decisive step 讓系統偏航」。

AgentLocate 先讓 LLM Judge 提出 agent-step 假設，再用多個 Evaluator 以 confidence-weighted voting 挑戰它，最後把 evaluator rationale / disagreement 變成 Judge refinement data。

Abstract p.1; §3 p.4-6

問題：multi-agent failure 的 root cause 常被 downstream symptom 蓋住

  • 角色分工、tool use、shared state 讓單一小錯沿著後續 prompt / tool outcome / coordination pattern 擴散。
  • 真正關鍵的是 which agent 與 earliest decisive step，不是最後哪個 agent 輸出錯答案。
  • 越長的 trajectory，越容易把「錯誤首次變可見的位置」誤認為「錯誤來源」。
這篇的核心觀點：debug agent 系統時，要找 root cause，不是追最後一個症狀。
§1 p.1-2; §5 p.9

為什麼現有方法不夠穩？兩條路都會漏掉 causal structure

Counterfactual replay / AgenTracer

  • 理論上更接近 causal testing。
  • 但改一個 step 會改後續 prompts、tool outcomes、coordination patterns。
  • 因此在 multi-agent setting 下，失敗誘因不一定能被穩定釘住。

Template / taxonomy / AEGIS

  • 用 curated error patterns 做 responsibility matching。
  • 對 emergent reasoning drift 與 coordination breakdown 覆蓋不足。
  • poisoning forensics 更假設 adversarial trace，與 benign failure mismatch。
§1 p.2; §5 p.9

形式化：輸出不是一個 label，而是一個 agent-step pair

Trajectory: τ = (s0, a0, s1, a1, ..., sT)
Goal: infer (i*(τ), t*(τ)) = responsible agent + earliest decisive step
  • scheduler ρ(t) 決定第 t 步 acting agent；若 corrected action at t* 能反轉 failure outcome，該步就是 decisive。
  • 這使 localization 同時要求 Who 與 When：只找出 agent 不夠，找太晚的 step 也不夠。
§2.1 p.3; Definition 1 §3 p.4
Method spine

Judge → Evaluators → Fine-tuned Judge

AgentLocate 把一次性 attribution 拆成「假設、驗證、吸收批評」三個動作；每個 failed trajectory 都變成下一輪 Judge 的訓練訊號。

§3 p.4-6

Stage 1 · Judge hypothesis generation

All-at-once

  • Judge 一次讀完整 failed trajectory。
  • 直接輸出 (î, t̂)。
  • 適合 trajectory 較短或需要全局脈絡。

Step-by-step

  • Judge 每次只看 growing prefix。
  • 若當前 action 已滿足 decisive-error criterion，立即輸出 (ρ(t), t)。
  • 更貼近「最早 decisive point」的搜尋。
(î, t̂) = Judge(p, τ, q)
§3.1 p.4-5

Stage 2 · Evaluator verification：讓多個觀點挑戰 Judge

  • 每個 Evaluator 看到 Judge 的候選 `(î,t̂)`，但可以同意或反駁。
  • prompt styles 包含 base、concise、evidence-focused，目的是得到互補 reasoning behavior。
  • 每個 evaluator 輸出候選 location、rationale 與 self-reported confidence。
Evaluatorh → (ih, th, rh, ch), where ch ∈ [0,1]
§3.2 p.5; Appendix D p.15-16

Aggregation：不是多數決，而是 confidence-weighted support

A(i,t) = Σh ch · 1{(ih,th) = (i,t)}
(ieval, teval) = argmax(i,t) A(i,t)
  • Evaluator rationale / confidence 不只是投票輔助；後面會成為 Judge refinement 的 supervision。
  • 這一步降低單一 model run 或單一 prompt style 的偶然偏差。
§3.2 p.5

Stage 3 · Adaptive Judge improvement

  • 如果 Judge 與 evaluator aggregate 不一致，差異本身就是「Judge reasoning 哪裡偏離 decisive criterion」的訊號。
  • 即使一致，evaluator rationale 也補上 missing evidence / causal link。
  • 每個 failed query 形成一筆 richer training instance，而非只保留 final label。
zq = {τ, q, î, t̂, {ih,th,rh,ch}h, ieval, teval}
§3.3 p.5-6

實驗設計：兩個 failure localization benchmarks + 多模型檢驗

  1. Who&When：Algorithm-Generated 126 cases / 191 agents / avg 8.6 steps；Hand-Crafted 58 cases / avg 50 steps。
  2. Aegis-Bench：9,533 failure trajectories、6 task domains、6 agent frameworks、24,843 injected error instances。
  3. Models：Qwen-7B、Llama-8B、Mistral-7B、GPT-4o；Who&When 50/10/40 train/val/test。
  4. Metrics：Who&When 用 agent-level + step-level；Aegis-Bench 用 agent-level + pair-level。

作者自己的診斷：visibility bias 仍是未解問題

  • 最常被 mislocalized 的 agent 往往是 Verification_Expert、WebSurfer 這類後段驗證 / 檢索角色。
  • 原因不是它們一定最常犯 root error，而是 upstream mistakes 常在這裡才「變得可見」。
  • 未來需要更顯式 modeling error propagation：分離 true upstream source 與 failure surface。
§5 p.9; Figure 5 p.23

局限：這不是「已解決 agent debugging」

  1. Exact step 仍難：Hand-Crafted 長 trajectory 下 Qwen all-at-once step-level 只有 16.67%，雖然仍優於多數 baseline。
  2. Error-mode pair 不完全對齊目標：Aegis-Bench pair-level margin 較小，作者也說 AgentLocate 主設計目標是 Who/When，不是 taxonomy classification。
  3. Evaluator model 差異有限：換 evaluator model 的改善 modest，表示方法增益主要不是「換更強模型」本身。
  4. Code release 尚未落地：paper 只寫 code will be released upon acceptance。
Table 2 p.20; Appendix G p.18; Table 9 p.23; §4.1 p.7

工程啟發：Agent framework 應該保存可被 challenge 的 failure trace

  • 日誌格式要保留 acting agent、step index、state / tool result、query context，否則 `(i,t)` localization 沒有落點。
  • Debug pipeline 不宜只做一次 LLM judge；至少要讓不同 prompt/perspective 對同一候選做 independent critique。
  • 把 disagreement / rationale 存成可訓練資料，比只存 final correct label 更有價值。
  • 對 production agent，這可以變成「post-mortem → evaluator review → judge adapter update」的閉環。
這篇最值得帶走的不是某個 benchmark 數字，而是 agent observability 與 refinement data schema 的設計方向。
§3 p.4-6; §5 p.9

Appendix highlight · 一次性 attribution 為何不夠

The core motivation is that a single attribution pass is often brittle for long, tool-mediated trajectories, so localization should be treated as a process that can be challenged, verified, and improved rather than a one-shot decision.

核心動機是：對長、含工具的 trajectory，單次歸因常常脆弱；failure localization 應被視為可挑戰、可驗證、可改進的流程，而不是一次性判決。§1 p.2

今天的 takeaway

Agent debugging 的下一步不是更多 logging 而已，而是能把 failure trace 變成可驗證、可反駁、可訓練的 supervision。
