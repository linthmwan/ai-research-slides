PairCoder++: Verified Pair Programming for Code-Driven Artifacts

Junhao Chen, Xiang Li, Mingjin Chen, Boran Zhang, Henghaofan Zhang, Yibin Xu, Yuehan Cui, Fangsheng Weng, Fei Ma, Qi Tian, Ruqi Huang, Hao Zhao
ACL 2026 acceptedarXiv 2607.01883Announced 2026-07-02
Source: arXiv:2607.01883, cover p.1; arXiv comment accepted by ACL 2026
TL;DR

把 toolchain 變成第二位工程師能引用的證據。

PairCoder 用 Driver 寫程式、Navigator 看 compiler / executor / renderer 證據審查；錯誤持續時，診斷者直接接手修復。

Abstract p.1; §3 p.4-7

問題不是「模型不會寫 code」而是 artifact 邊界不可見

  • Code-driven generation 的輸出是 程式渲染出的 artifact：TikZ、chart、SVG、CAD、Blender、React、Verilog。
  • 單次 inference 常在 compiler / renderer / simulator 邊界失敗：不編譯、不執行，或渲染結果離需求很遠。
  • 這些失敗訊號其實很有用，但 baseline 在寫 code 時看不到它們。
關鍵轉折：不要只要求 LLM 自我修正，要讓外部 toolchain 開口。
§1 p.2

既有方法卡在兩端：team 太貴，self-correction 太單薄

Multi-agent team / pipelines

  • 角色分工能改善 correctness。
  • 但常付出接近 order-of-magnitude token overhead。
  • 而且很多系統是 domain-specific engineering。

Single-model self-correction

  • 較便宜，能利用 runtime feedback。
  • 但仍繼承同一模型視角的 blind spots。
  • 缺少 grounded independent reviewer。
PairCoder 站在交會點：最小合作單位，卻把 reviewer 綁到 verification evidence 上。
§1 p.2; §2.3 p.3

三個貢獻

  1. PairCoder protocol：Driver / Navigator、verification-grounded review、error-triggered role switching。
  2. 跨 domain 證據：17 benchmarks、7 models、3 vendors，涵蓋 code synthesis 到 multimodal artifact generation。
  3. 機制規律：收益集中在 oracle informative 且 baseline 有 headroom 的任務；oracle 弱或飽和時 tie 或小幅退步。
§1 p.2; §4.1-4.3 p.7-9

框架：Driver 產生，Navigator 依證據審查

Figure 2 p.4; §3.1 p.4

核心 protocol：只允許兩種 verdict

Ct = Fdriver(Idri, Q, Mt-1)
  • Driver 第一輪只看 task Q。
  • 後續收到上一版 code、Navigator review 與同一份 evidence。
  • 沒有 fine-tuning；全靠 role prompt 和 shared history。
Rt = [NOERROR] 或 REVISE(Δt)
  • Navigator 必須引用錯誤行、違反需求與具體修法。
  • 若 evidence 已通過且找不到 definite fault，就必須接受。
  • 這避免對正確程式做 style-level churn。
§3.2 p.4-5
Mechanism crown page

第二視角之所以有用，是因為它看到第一位看不到的證據。

ψt 包含 compile / execute predicate、diagnostic、可選測試、以及 current render vs target。這讓 critique 從「我覺得」變成可定位、可修復的工程訊號。

§3.2 p.5

Verification evidence ψt：唯一 domain-specific 部分

  • Compiler / renderer / simulator：LaTeX、CadQuery、matplotlib、SVG rasterizer、Blender、RTL simulator。
  • 若 benchmark 有 specification，Navigator 可以先寫 test 再 review。
  • Image-conditioned 任務會給 Navigator 看 current artifact 與 target 並列。
  • Loop 其他部分在 17 個 benchmark 間保持一致。
這是一個「外掛 verifier」的 agent pattern，而不是新的模型架構。
§3.2 p.5; §3.3 p.5

Role switching：診斷者接手鍵盤

  • Default policy：每次 REVISE 後立即切換角色。
  • 直覺：剛指出 fault 的 Navigator 最知道該怎麼修。
  • Self-Mirror prompt 在每次 generation / review 前重新聲明目前角色，維持 role consistency。
§3.4 p.6-7

Budget 用完時，不回傳最後一個候選，而是回傳最好候選

C* = arg maxCi in Mt Quality(Ci)
  • 若 `[NOERROR]`，直接回傳目前 candidate。
  • 若超過 budget T，Quality 先看 verification 是否 pass，再看 continuous score，最後看 recency。
  • 這讓未解決的 review debate 降級成 safe fallback，而不是最後一次嘗試。
§3.4 p.6

實驗設定：17 benchmarks x 7 models x 3 vendors

  • Program synthesis：LiveCodeBench、BigCodeBench、DS-1000。
  • Multilingual / Web / Hardware：HumanEval-X、WebApp1K、VerilogEval、RTLLM。
  • Code-driven artifacts：DaTikZ、Plot2Code、PandasPlotBench、ChartMimic、StarVector、GenCAD-Code、3DCodeBench、P3D-Bench。
  • Models：gpt-5.4-mini / 5.4 / 5.5、Doubao 兩款、DeepSeek 兩款；main grid 關閉 thinking。

跨 vendor 也成立，但 oracle 弱時會露出邊界

  • DaTikZ 在四個非 OpenAI models 上皆提升 10-30 compile points。
  • 3DCodeBench 在四個中有三個提升 22-33 executability points。

PairCoder 不是萬能；它放大的是 verifier 的資訊量。

§4.5 p.10

Ablation-like analysis：立即切換角色最好或並列最好

  • Immediate switch 在四個 benchmark 上 best 或 tied-best，且從不是 worst。
  • HumanEval-X C++：immediate 0.427；after two errors / no switch 都是 0.390。
  • 解釋：讓診斷者留在 reviewer seat，會浪費它剛形成的 fault model。
§4.6 p.10-11

Reasoning effort：強 reasoning 也不能替代 verifier

BigCodeBench

  • No thinking：0.450 -> 0.458 (+0.8 pp)。
  • High reasoning：0.425 -> 0.475 (+5.0 pp)。
  • 更強 Driver 留下的錯，較容易被 doctest evidence 抓到。

DS-1000

  • Thinking 把 baseline 0.250 拉到 0.310。
  • 但 PairCoder 0.310 -> 0.300，近似噪音。
  • 因 oracle 只有 signature execution，不能裁判功能正確性。
§4.7 p.11

Multimodal artifact：硬失敗變少，已能渲染時再修細節

Figure 8 p.12; §5 p.11-12
Interpretation

PairCoder 的本質是把「生成」改造成「有驗證證據的工程 review」。

對 AI 圖像 / 3D / CAD 工程來說，這意味著 prompt chaining 應該圍繞 renderer diagnostics、visual diff、geometry tests 設計，而不是只加更多 reflective text。

§5 p.11-12; Conclusion p.13

限制：高成本與 oracle 依賴

  • 平均約 7x token cost，依 benchmark 為 2.9x-9.2x；部署時 latency 和 budget 是實際限制。
  • 若 verifier 只提供弱訊號，Navigator 可能 model-bound，甚至 churn 已可用的 code。
  • 方法目前是 inference-time protocol；沒有證明能取代重試、fine-tuning 或 domain-specific evaluator design。
工程上要先問：我能不能建一個可靠 oracle？若不能，pair programming 可能只是多花 token。
Limitations p.13; §4.3 p.9; §4.7 p.11

落地 recipe：把 PairCoder pattern 搬到你的生成系統

  1. 把 artifact 產生器包成 deterministic toolchain：compile / render / simulate / test。
  2. 把 diagnostics 和 rendered output 明確餵給 reviewer agent，而不是只餵上一輪文字。
  3. review verdict 要二值化：accept 或 concrete revise；禁止 vague improvement。
  4. 讓 diagnosing agent 接手下一輪修復；budget 用完回傳 best verified candidate。
